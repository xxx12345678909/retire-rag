## 二、逐一分析

### 2.1 概念正确性 ✅ — "关键词扩展"方向对

文档的核心思想没错：

> RAG 只能回答"已有知识" → 通过关键词扩展 + 自动采集 → 让知识库"持续增长"

这在 RAG 工程中就叫 **Query Expansion** + **Automated Corpus Building**，确实是提升 RAG 覆盖率的有效手段。当前项目的手动上传 PDF 模式确实是瓶颈。

**但注意：** 文档把"关键词扩展"和"语料采集"绑在一起讲，概念上混淆了两个独立模块：
- **Query Expansion**：在线阶段，把用户问题扩展为多个检索词，提升召回率（不改知识库）
- **Corpus Acquisition**：离线阶段，自动搜集新文档扩充知识库

这两个应该分开设计。文档把它们混为一个 pipeline，会导致职责不清。

### 2.2 工程可行性 ❌ — 4 个硬伤

#### 硬伤 1：Google 搜索在中国不可用

```python
url = f"https://www.google.com/search?q={keyword}"
res = requests.get(url)   # ❌ 在中国大陆直接访问 Google 返回 404/超时
```

且即使能访问，`requests.get` 拿到的 HTML 不含搜索结果（Google 前端是 JS 渲染的），需要用 Selenium/Playwright 或付费 SERP API。

#### 硬伤 2：爬虫代码是"教学伪代码"

```python
soup = BeautifulSoup(res.text, "html.parser")
for a in soup.find_all("a"):
    href = a.get("href")
    if href:
        links.append(href)   # ❌ 会抓到导航链接、广告、footer，不是搜索结果
```

实际需要的：
- 反爬：User-Agent、IP 代理池、请求频率控制
- 解析：按搜索引擎 DOM 结构提取真实结果链接（每个引擎结构不同）
- 过滤：去重、去广告、去非目标域名
- 合规：`robots.txt` 检查

#### 硬伤 3：与现有项目技术栈冲突

| 文档建议 | 现有项目 | 冲突 |
|---|---|---|
| `pdfplumber` 解析 PDF | 已有 `PyPDF` + `file_handler.py` | 引入新依赖，且现有 `pdf_loader()` 可复用 |
| 写入 `data/{keyword}.txt` | 知识库目录为 `data/policy/` 等 4 个分类目录 | 直接写入裸目录会破坏现有的 MD5 去重和向量化流程 |
| 新建 `data_pipeline/` 目录 | 现有包结构 `retire_rag/` | 应该放在 `retire_rag/` 内部，复用 `utils/`、`rag/` 模块 |

#### 硬伤 4：文件命名不规范

文档中文件名 `data/{keyword}.txt` 用关键词做文件名——中文关键词含特殊字符（如 `site:gov.cn`），在 Windows 上会导致文件创建失败。

### 2.3 与现有项目契合度 ❌

文档完全没有融入现有系统：

- **不知道已有的 4 个知识库分类**（policy/service/health/platform）——采集的语料该归于哪个库？
- **不知道已有的 Chroma + MD5 去重体系**——采集后如何自动向量化？
- **不知道已有的 `data_loader_cleaner.py`**——已有全文清洗逻辑，文档又写了一套简化版
- **不知道已有 `VectorStoreService.load_document()`**——重复造轮子

这导致如果按文档开发，会产出一个**与现有项目不兼容的独立模块**，需要大量胶水代码对接。

## 四、如果要落地开发，建议的修正方案

### 4.1 先做 Query Expansion（不爬虫）

**最有价值且风险最低的第一步**：在现有 ChatService 的 RAG 检索前加入关键词扩展。

```python
# 在 chat_service.py 或 rag_service.py 中
class QueryExpander:
    def expand(self, query: str, kb: str) -> list[str]:
        """输入用户问题，输出 3-5 个扩展检索词"""
        prompt = f"将以下养老相关问题扩展为3-5个检索关键词：{query}"
        # 调用现有 chat_model 生成
        keywords = chat_model.invoke(prompt)
        return clean_keywords(keywords)
```

然后把多个检索词的召回结果合并去重，再拼装上下文。**不改知识库，只改检索层**，改动小、收益直接。

### 4.2 再做自动采集（离线异步任务）

爬虫部分应该是一个**独立的异步后台任务**，不应嵌入 ChatService 主流程：

```text
Celery / APScheduler 定时任务
  → KeywordExpander (基于 seed + 日志中未覆盖的提问)
  → URL Discoverer (Bing API / 政府网站RSS)
  → PDF/HTML Downloader
  → 接入现有 VectorStoreService.load_document()
```

关键技术选型：
- 搜索 API：**Bing Web Search API**（中国可用，有免费额度）替代 Google
- PDF 解析：**复用现有 `file_handler.pdf_loader()`**，不要引入 pdfplumber
- 存储：写入 `data/{kb}/` 对应目录，自动触发现有 MD5 去重 + 向量化流程
- 合规：只爬 gov.cn 等政府公开网站，遵守 robots.txt

### 4.3 采纳文档中的合理概念

| 文档概念 | 如何用 | 放在哪 |
|---|---|---|
| Keyword Expansion Engine | 改造为 QueryExpander 类 | `retire_rag/rag/query_expander.py` |
| `clean_keywords()` | 直接复用 | 同上 |
| 数据优先级 | 采集时按优先级排序 URL | 采集模块配置 |
| `filter_bad()` | 合并到 `DataCleaner` | `retire_rag/data_loader_cleaner.py` |
| 去重（hash） | 已有 MD5 体系，无需重复实现 | 无需动 |

## 五、总结

| 维度 | 结论 |
|---|---|
| **文档本身可作为开发依据吗？** | ❌ 不能。是概念草图，不是可执行的开发规范。 |
| **核心思路有价值吗？** | ✅ 有。关键词扩展 + 自动语料采集是 RAG 系统进化的正确方向。 |
| **值得在当前项目实现吗？** | 分步走：**先做 Query Expansion（高价值低风险），爬虫采集暂缓**（工程量大、合规风险高、现有手动上传已可用）。 |
