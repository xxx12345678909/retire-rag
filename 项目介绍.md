# 智慧养老AI助手 — 项目介绍与实现详解

> 基于 RAG（Retrieval-Augmented Generation）技术的智慧养老知识问答平台，面向老年人及家属提供政策解读、服务导办、健康科普、平台操作四大场景的智能化咨询。

---

## 一、项目概述

### 1.1 项目背景

随着中国人口老龄化加速（2025年60岁以上人口突破3亿），养老政策密集出台、养老服务形式日趋多元，但老年群体及其家属仍面临三大痛点：

- **政策信息获取难**：高龄补贴、长护险、养老金调整等政策分散于多部门文件，普通民众难以快速找到匹配自身条件的政策
- **服务选择成本高**：养老机构、居家服务种类繁多，缺乏统一的筛选和推荐渠道
- **健康知识真伪难辨**：网络上的老年健康信息质量参差不齐，且大模型存在"幻觉"风险

本项目以2024-2025年国务院及人社部、财政部等部门发布的真实养老政策文件为核心数据源，构建了一套**多知识库 RAG 系统**，确保回答内容有据可查、来源可溯。

### 1.2 核心目标

| 目标 | 说明 |
|---|---|
| **政策智能匹配** | 根据老人年龄、户籍、社保状况，自动检索匹配可享受的福利政策 |
| **服务精准导办** | 根据护理需求推荐养老机构或居家服务，引导预约流程 |
| **健康科普问答** | 基于权威科普资料回答老年常见病护理问题，强制附加免责声明 |
| **安全边界控制** | 医疗诊断类问题自动拒答，非法/违规请求拒绝处理 |

### 1.3 数据基础

知识库包含 **5份国务院部委真实政策文件 PDF** 及 **4份结构化 TXT 知识手册**：

- 2025年第6号国务院公报：全面实施个人养老金制度
- 2025年第26号国务院公报：领取个人养老金有关问题
- 两部委关于调整2025年退休人员基本养老金的通知
- 三部委关于大龄领取失业保险金人员参加企业职工基本养老保险的通知
- 养老政策法规汇编（结构化 TXT）
- 养老服务项目手册、老年健康科普手册、平台操作指南

## 三、核心模块实现详解

### 3.1 统一对话入口（`app.py`）

```python
POST /api/v1/chat
{
    "query": "我爸75岁农村户口能领什么补贴",
    "session_id": "user_abc123",
    "use_agent": false          // 可选：true = ReAct Agent, false = ChatService
}
```

**设计要点：**

- **双模式切换**：`use_agent` 参数控制路由到 ChatService（规则路由）或 ReactAgent（LLM 自主决策），Agent 导入失败时自动降级，不阻塞应用启动
- **启动时预热**：通过 FastAPI `lifespan` 机制在应用启动时自动加载全部 4 个知识库，首次请求即可检索
- **统一响应格式**：所有回答包含 `answer`（正文）、`intent`（意图类别）、`kb`（知识库）、`sources`（参考来源）、`disclaimer`（风险声明）五个字段
- **问答日志**：每次对话自动记录，支持按知识库筛选、按质量标注反馈（1/-1），闭环优化

### 3.2 意图识别与业务路由（`ChatService`）

ChatService 是整个系统的**业务大脑**，实现了从自然语言到结构化回答的完整链路。

#### 3.2.1 意图识别机制

采用**关键词匹配 + 诊断拒答优先**的三级策略：

```
用户输入
  │
  ├─ 匹配诊断拒答模式（"我是什么病"、"开药"、"处方"...）
  │   └─ 命中 → 自动拒答，提示就医
  │
  ├─ 匹配健康关键词（"糖尿病"、"高血压"、"吃什么"...）
  │   └─ 命中 → intent = health
  │
  ├─ 匹配政策关键词（"补贴"、"高龄"、"养老金"、"长护险"...）
  │   └─ 命中 → intent = policy
  │
  ├─ 匹配服务关键词（"养老院"、"失能"、"预约"、"收费"...）
  │   └─ 命中 → intent = service
  │
  └─ 均未命中 → intent = general（默认走平台操作库）
```

**设计决策**：选择关键词匹配（而非 LLM 分类）的原因是——在养老场景下，关键词覆盖率高、响应延迟为零（无需 API 调用）、结果可解释、不受 LLM 幻觉影响。该设计对齐了 Alibaba Tongyi 官方推荐的"方案A（稳定优先）"策略。

#### 3.2.2 四大业务流

每个业务流遵循相同的模式：**RAG 检索 → 上下文拼装 → 模板回答 → 附加声明**，但各有差异化处理：

| 业务流 | 检索 KB | 特殊处理 | 免责声明 |
|---|---|---|---|
| `policy_flow` | 政策法规库 | 空结果时引导用户咨询民政部门/拨打12349热线 | "具体以当地民政部门最新政策为准" |
| `service_flow` | 养老服务库 | 空结果时引导浏览服务页面/拨打客服热线 | 无 |
| `health_flow` | 健康科普库 | 空结果时建议咨询专业医生 | **强制**附加医疗免责声明 |
| `general_flow` | 平台操作库 | 空结果时给出换方式提问的引导 | 无 |

#### 3.2.3 多轮对话记忆

```python
# 取最近 3 轮历史拼入上下文查询
def _build_query(self, session_id, query):
    history = self._sessions.get(session_id, [])
    recent = history[-3:]
    context_parts = []
    for h in recent:
        context_parts.append(f"用户：{h['query']}")
        context_parts.append(f"助手：{h['answer'][:200]}")  # 截断防止过长
    return "\n".join(context_parts + [f"用户：{query}"])
```

**关键实现细节**：
- 每会话保留最近 **10 轮**历史，超出自动淘汰
- 构建上下文查询时取最近 **3 轮**（平衡上下文长度与相关性）
- 历史回答截断至 **200 字符**（防止 token 膨胀）
- 意图识别基于含历史的 `contextual_query` 执行，支持多轮追问（如"申请条件是什么" → "需要什么材料"）

### 3.3 RAG 检索引擎（`RagService` + `VectorStoreService`）

#### 3.3.1 多知识库隔离架构

4 个知识库对应 4 个独立的 Chroma Collection，物理隔离，互不干扰：

```yaml
# chroma.yml
collections:
  policy:   { name: retire_policy,   data_path: data/policy }
  service:  { name: retire_service,  data_path: data/service }
  health:   { name: retire_health,   data_path: data/health }
  platform: { name: retire_platform, data_path: data/platform }
```

`VectorStoreService.__init__()` 在同一个 `persist_directory` 下初始化 4 个 `Chroma` 实例，每个绑定独立的 Collection。

#### 3.3.2 文本分片策略

```python
RecursiveCharacterTextSplitter(
    chunk_size=300,       # 中文约150字一个分片
    chunk_overlap=50,     # 25字重叠，保证语义连贯
    separators=["\n\n", "\n", "。", ".", "?", "？", "!", "！", " ", ""],
    length_function=len,
)
```

**设计要点**：分隔符优先级从粗到细（段落 → 换行 → 中文句号 → 英文句号 → 问号 → 感叹号 → 空格 → 字符），确保中文政策文档按语义边界切分，而非生硬截断。`chunk_size=300` 和 `overlap=50` 是针对中文政策文件（平均每句 30-50 字）的调优结果——一条典型政策条款（约 100-200 字）落在 1-2 个 chunk 内，不会被打散。

#### 3.3.3 MD5 去重机制

```python
# 每个知识库维护独立的 MD5 跟踪文件
md5_policy.txt    # 记录已加载政策的文件哈希
md5_service.txt   # 记录已加载服务的文件哈希
...
```

加载流程：
1. 扫描 `data/{kb}/` 目录下所有 `.txt` / `.pdf` 文件
2. 计算每个文件的 MD5 哈希
3. 与对应 `md5_{kb}.txt` 对比：已存在则跳过，不存在则加载并写入记录

**价值**：避免应用重启时重复向量化已处理文档，大幅缩短启动时间。

#### 3.3.4 文档来源检索

```python
def retrieve_context(self, query, kb="platform"):
    docs = self.search(query, kb)
    context = "\n---\n".join([
        f"[来源：{doc.metadata.get('source', '未知文档')}]\n{doc.page_content}"
        for doc in docs
    ])
    return context
```

检索结果以 `top-k=3` 返回，每个文档片段标注 `[来源：文档名]`，实现**回答溯源**能力——用户可以看到每条信息来自哪份文件。

### 3.4 ReAct Agent 模式（`ReactAgent`）

当用户设置 `use_agent=true` 时，系统切换到 LangGraph ReAct Agent 模式。

#### 3.4.1 ReAct 循环

```
Thought（思考）→ Action（工具调用）→ Observation（观察结果）
    ↑                                              ↓
    └──────────── 循环（最多3次工具调用）←────────────┘
                              ↓
                    Final Answer（最终回答）
```

#### 3.4.2 工具集

Agent 拥有 6 个工具（`@tool` 装饰器注册为 LangChain Tool）：

| 工具 | 功能 | 入参 |
|---|---|---|
| `rag_retrieve_policy` | 检索政策法规库 | query: str |
| `rag_retrieve_service` | 检索养老服务库 | query: str |
| `rag_retrieve_health` | 检索健康科普库 | query: str |
| `rag_retrieve_platform` | 检索平台操作库 | query: str |
| `get_weather` | 获取城市天气 | city: str |
| `get_user_location` | 获取用户城市 | 无 |

**关键实现细节**：
- 工具函数名和 docstring 就是 Agent 的"说明书"——LangGraph 会将 docstring 传给 LLM，LLM 据此决定调用哪个工具
- `get_weather` 和 `get_user_location` 是天气/位置感知能力的演示占位（硬编码5个城市数据，随机返回深圳/合肥/杭州）
- 4 个 RAG 工具的 docstring 明确标注了使用场景（"用户询问补贴申请条件时调用"），避免 Agent 滥用或选错工具

#### 3.4.3 优雅降级

```python
def __init__(self):
    try:
        from langgraph.prebuilt import create_react_agent
    except ImportError:
        logger.warning("create_react_agent 不可用，Agent 模式已禁用")
        self.agent = None
        return
```

LangGraph 版本兼容问题在导入阶段被捕获，Agent 置为 `None`，ChatService 模式不受影响。应用启动时全局 `agent` 变量做同层保护。

### 3.5 模型工厂（`ModelFactory`）

采用抽象工厂模式管理 LLM 和 Embedding 模型：

```python
class ChatModelFactory(BaseModelFactory):
    def generator(self):
        return ChatTongyi(
            model="qwen3.7-plus",
            top_p=0.8,
            temperature=0.1,       # 低温度保证回答稳定性
            api_key=os.getenv("DASHSCOPE_API_KEY")
        )

class EmbeddingsFactory(BaseModelFactory):
    def generator(self):
        return DashScopeEmbeddings(
            model="text-embedding-v3",
            dashscope_api_key=os.getenv("DASHSCOPE_API_KEY")
        )
```

**参数选择理由**：
- `temperature=0.1`：养老政策问答需要事实准确性而非创造性，低温度减少随机性
- `top_p=0.8`：保留一定候选词范围，避免生成过于机械
- `text-embedding-v3`：DashScope 最新的嵌入模型，中文语义检索效果优于 v2

### 3.6 安全边界与合规控制

| 控制点 | 实现方式 | 效果 |
|---|---|---|
| 医疗诊断拒答 | 14 个拒答模式关键词匹配（"确诊"、"开药"、"处方"、"吃什么药"等） | 命中后直接返回拒答文案，不触发任何 RAG 检索 |
| 健康免责声明 | `health_flow()` 返回时强制拼接 ⚠️ 免责声明 | 每一条健康回答都附带 |
| 政策免责声明 | `policy_flow()` 返回时拼接 📌 政策提示 | 每一条政策回答都附带 |
| 来源引用 | 所有回答末尾附带 "📋 参考来源" 板块 | 增强回答可信度，符合 RAG 技术特征 |
| 有害内容拒绝 | Agent 系统提示词中明确要求拒绝违法违规问题 | LLM 层面的第二道防线 |

### 3.7 环境兼容性补丁

```python
# packaging 元数据修复补丁
import importlib.metadata as _importlib_metadata

def _patched_version(package_name: str) -> str:
    v = _orig_version(package_name)
    if v is None and package_name == "packaging":
        return "26.2"           # 硬修复被 23.1 空目录遮蔽的版本号
    return v

_importlib_metadata.version = _patched_version
```

**问题背景**：Windows 环境中存在一个空的 `packaging-23.1.dist-info` 目录遮蔽了有效的 `packaging-26.2.dist-info`，导致 `importlib.metadata.version('packaging')` 返回 `None`，进而使 `transformers` 和 `langchain_text_splitters` 的版本检查失败。此补丁通过在 `transformers` 导入前拦截 `version()` 调用，将 `packaging` 的版本硬编码为 `26.2`，确保所有依赖正常加载。

此补丁在 `app.py` 和 `vector_store.py` 中**各执行一次**（谁先被导入谁先生效），因为 Python 模块系统保证同一进程内 `_importlib_metadata.version` 只被 monkey-patch 一次。

## 五、项目亮点与技术特征

### 5.1 面向评审的关键技术点

1. **多知识库 RAG 架构**：4 个独立 Chroma Collection + 意图路由，实现"一个系统、四条管线"的领域隔离检索，区别于单一知识库的简单 RAG 实现

2. **双模式架构**：ChatService（规则路由，零延迟、确定性高）和 ReactAgent（LLM 自主决策，灵活智能）可通过参数切换，体现了系统工程思维

3. **MD5 去重流水线**：每个知识库独立维护文件哈希记录，解决"重启后重复向量化"的工程问题，是生产级 RAG 系统的必要组件

4. **安全边界三层控制**：关键词拒答（第一层）→ 业务流强制免责声明（第二层）→ Agent 系统提示词约束（第三层），医疗场景下尤为关键

5. **回答溯源引用**：每个回答附带 `[来源：文档名]` 标注和 `sources` 结构字段，符合 RAG 技术范式，提升回答可信度

6. **多轮对话上下文管理**：取最近 3 轮历史拼接 + 回答截断 200 字 + 会话 10 轮上限，简单有效的工程方案

7. **环境兼容性修复**：针对 Windows 下 `packaging` 版本检测异常的 monkey-patch，展示了实际开发中的问题定位和修复能力

8. **优雅降级设计**：Agent 模块导入失败不阻塞应用、未知知识库自动 fallback 到 platform、空检索结果返回引导文案——全线防御异常路径

9. **配置分离**：敏感的 API Key 存放在 `.env`，业务配置在 `config/*.yml`，代码逻辑在 Python 模块，三层分离符合安全最佳实践

10. **适老化前端交互**：标准版/适老版切换、大字体高对比度、语音输入预留接口，体现了对目标用户群体（老年人）的深度关注

### 5.2 可扩展方向（供评审参考）

- **结构化信息提取**：在 `policy_flow` 中接入 LLM 提取用户年龄/户籍/社保等结构化条件，与政策库做精确匹配（当前为关键词检索）
- **向量库升级**：从 Chroma 迁移至 Milvus/Qdrant，支持更大规模知识库和混合检索（dense + sparse）
- **反馈闭环**：基于 `_qa_logs` 中的标注数据，定期用低分问答反哺知识库（补充缺失文档、调整分片策略）
- **语音交互**：前端预留了语音输入/播报的 UI，对接 Web Speech API 或第三方 ASR/TTS 服务
- **持久化存储**：问答日志和会话历史从内存迁移到 SQLite/PostgreSQL，支持生产环境部署
- **缓存层**：常见问题（FAQ）加入 Redis 缓存，减少重复 RAG 检索

## 七、快速启动

```bash
# 1. 安装依赖
cd retire_rag
pip install -r requirements.txt

# 2. 加载知识库（首次运行或新增文档后执行）
python data_loader_cleaner.py

# 3. 启动应用
python app.py
# 或：uvicorn app:app --reload --host 0.0.0.0 --port 8000

# 4. 访问
# 前端原型：   http://localhost:8000
# API 文档：   http://localhost:8000/docs
# 健康检查：   http://localhost:8000/health
# 知识库统计： http://localhost:8000/api/v1/kb/stats
```
