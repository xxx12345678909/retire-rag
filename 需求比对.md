# 需求文档 vs 当前实现 — 差距分析

> 标 `✅` 已实现 · `⚠️` 部分实现 · `❌` 未实现

---

## 一、AI 核心能力（需求第三、四节）

| # | 需求 | 状态 | 差距 |
|---|---|---|---|
| 1 | 自然语言问答，自动匹配知识库 | ✅ | ChatService 意图识别 + 4KB 路由 |
| 2 | 回答溯源引用（原文片段+来源文档名） | ✅ | `sources` 字段 + `[来源：xxx]` 标注 |
| 3 | 多轮对话记忆（上下文追问） | ⚠️ | 有，但在内存里，重启即丢；历史截断逻辑可用 |
| 4 | 拒答与边界控制（诊断拒答+免责声明） | ✅ | 14 个拒答模式 + health_flow 强制免责 |
| 5 | 政策智能匹配（提取年龄/户籍/社保 → 匹配福利） | ❌ | 当前只做 RAG 关键词检索，没有结构化信息提取和条件匹配 |
| 6 | 服务智能导办（需求描述 → 筛选机构 → 引导预约） | ❌ | 当前只做 RAG 检索服务库，没有机构 DB 查询、筛选、预约引导 |
| 7 | 健康科普问答（限制版RAG，避免幻觉） | ⚠️ | 有 RAG + 免责，但没有避免 LLM 幻觉的约束（当前是模板拼装不用 LLM 生成，天然无幻觉） |

## 五、优先级建议

### 🔴 第一优先（AI 核心能力，影响评审）

| # | 要做 | 工作量 | 理由 |
|---|---|---|---|
| 5 | **政策智能匹配** — 提取年龄/户籍/社保 | 1-2天 | 这是项目最核心的差异化功能，"政策匹配" ≠ "搜政策文档"，评审会看这个 |
| 6 | **服务智能导办** — 机构筛选+推荐 | 2-3天 | 同上，"服务导办"要看到推荐逻辑，不只是搜 FAQ |
| 9 | Word 文档上传 | 0.5天 | `python-docx` 提取文本，改动 `file_handler.py` 几行 |

### 🟡 第二优先（工程完善）

| # | 要做 | 工作量 | 理由 |
|---|---|---|---|
| 11 | 问答日志持久化（内存→SQLite） | 1天 | 日志是知识库优化的前提，重启就丢等于没做 |
| 3 | 多轮会话持久化 | 0.5天 | 同上 |
| 10 | 分片参数可配置 API | 0.5天 | `/admin/kb/chunk-config` 已有 chroma.yml，加个 API 即可 |

### 🟢 第三优先（锦上添花）

| # | 要做 | 工作量 |
|---|---|---|
| 17 | 语音输入/播报 | 1天（对接 Web Speech API 或 DashScope 语音） |
| 16 | 场景化入口 | 1天（前端加 `?kb=policy` 参数，后端按参数预选 KB） |
| 21-26 | 业务模块后端 | 2-4周（独立系统，非 RAG 范畴） |

---

> **核心结论**：AI 问答骨架 ✅ 齐全，缺的是 **#5 政策智能匹配**（结构化信息提取）和 **#6 服务智能导办**（DB 查询+推荐逻辑），这两个是"通用聊天机器人"和"养老业务系统"的分界线。
